# Tic-Tac-Toe (MCTS vs Random)

このベンチマークは、第4章で述べた「推論時（test-time）計算量スケーリング」を、**おもちゃ問題**で再現するための最小例です。

## 何を測るか

- **品質（quality）**: MCTSエージェントの勝率（vs ランダム）
- **推論時計算量のproxy**: 1手あたりのシミュレーション回数（`simsPerMove`）

## 実行

`experiments/run.md` を参照してください。

## 注意

- 三目並べは小規模で、実システムの複雑性とは異なります。ここでは「計算資源（proxy）を増やすと品質が改善しうる」ことを確認するのが目的です。
- 実務のLLMでは、トークン数/探索分岐/外部ツール呼び出し/wall-clock/コスト等の複合指標で扱ってください。
